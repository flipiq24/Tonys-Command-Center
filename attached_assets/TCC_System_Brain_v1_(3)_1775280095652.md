# 1. SYSTEM OVERVIEW

This is Tony Diaz's personal operating system for running FlipIQ and his daily life. It is a **sequential single-view command system** — ONE screen at a time, not a multi-panel dashboard. The system's job is to keep Tony focused on what matters most, protect his time, enforce accountability, and get smarter over time through a dynamic training log.

## Core Principles
- Tony has ADHD-driven scattered energy. This system CHANNELS that energy into the RIGHT thing at the RIGHT time.
- It does NOT overwhelm — it SEQUENCES.
- ONE view at a time. When it's email, the whole screen is email. When it's sales, the whole screen is sales. When it's calendar, the whole screen is calendar.
- Tony does NOT like to read. Keep everything visual, clickable, and minimal text.
- Tony speaks, not types. Voice-to-text is the primary input method.
- Every action the system learns from gets logged to the TRAINING LOG (Section 10).

## Tony's North Star
- FlipIQ: Every Acquisition Associate closes 2 deals/month.
- Tony's job is SALES. Everything else is secondary.
- Revenue target: $50K break-even → $100K Phase 1 → $250K Scale

## Tony's Scripture Anchors
- "But seek first the kingdom of God and his righteousness, and all these things will be added to you" — Matthew 6:33
- "Commit your work to the Lord, and your plans will be established" — Proverbs 16:3

## Tony's Personal Rules
- "Today, I will follow the plan I wrote when I was clear. Not the plan I invent when I'm afraid."
- "I do not need to build something massive to prove my greatness. I prove my greatness by obeying what is clear."
- "I do not substitute clarity for action. I execute. I stabilize. I serve."
- "My ego expands. God simplifies. Today, I choose simplicity."

## Architecture
- Sequential views controlled by a persistent header
- Each view takes the FULL SCREEN
- Header stays visible at all times with: Title, Date + LIVE TIME, Quote, and action buttons
- Data flows: Dashboard ↔ Claude API (MCP) ↔ Gmail, Calendar, Slack, Linear, AgentMail
- Persistent storage: Supabase (when on Replit) or window.storage (when artifact)
- Learning: All preferences saved to this MD file via Training Log

## Step 3: IMPORTANT EMAILS (Popup)

### Purpose
Show Tony emails that need his attention RIGHT NOW. Train over time to get smarter.

### How It Works
1. Cowork pre-sorts emails overnight using IMPORTANCE RULES (Section 7)
2. On system load, important emails appear as a popup/overlay
3. Each email shows:
   - **FROM** (sender name)
   - **SUBJECT** (email subject line)
   - **WHY IT MATTERS** (one sentence AI-generated explanation of why this is important)

### Email Labels (NOT "Action Required")
- **Important Emails** — emails that need Tony's response or decision. HAVE reply capability.
- **FYI** — emails Tony should know about. NO reply capability. Just information.

### Actions Available on Important Emails
| Action | What Happens |
|--------|-------------|
| **Snooze** | Flexible: 1 hour, 2 hours, tomorrow morning, next week, custom time. Email disappears until snooze expires, then returns. |
| **Suggest Reply** | System drafts a reply based on context → Tony reviews → approves → draft goes to Gmail |
| **Speak/Type Reply** | Tony dictates or types reply → draft goes to Gmail |
| **Thumbs Up 👍** | Confirms: "Yes, this was correctly flagged as important." System learns. Logged to Training Log. |
| **Thumbs Down 👎** | Tony says "Not important" → system asks WHY → Tony explains → reason logged to Training Log. System learns to not flag similar emails. |

### Actions Available on FYI Emails
- Read only. NO reply button. NO snooze. Just information.

### After Popup is Dismissed
- Unresolved important emails (not replied to, not snoozed) become a RED BADGE NUMBER on the Gmail button in the header
- When Tony clicks the Gmail button in the header → full-screen "All Emails" view showing:
  - Important Emails section (with reply/snooze)
  - FYI section (read only)
  - Notification count of unresolved items
- Once Tony closes the all-emails view, only the badge count remains

### New Email Alerts (During Use)
- If a new important email arrives while Tony is using the system → popup notification
- If not important → no interruption, just updates the badge count

## Step 4: SCHEDULE (Full Screen)

### Purpose
Show Tony his day. Smart time routing. Protect mornings for calls.

### What It Shows
- Current date and LIVE CLOCK
- All REAL meetings (events with other attendees or video/conference links)
- Calendar task-notes listed SEPARATELY (Tony's self-created reminder entries)
- For each real meeting:
  - Time
  - Title
  - Who is attending
  - Context from LAST interaction: "This is your 2nd meeting with this person. Last time you discussed [X]. You said you would [Y]."
  - Meeting notes history from previous meetings with this person
  - Location (if any)
  - Flags: "Stop attending" or "Hand to Ramy" (from OAP v4 rules)

### Smart Time Routing
After Tony acknowledges the schedule, the system evaluates his available time:

| Available Time | System Response |
|---------------|----------------|
| Under 15 min before next meeting | "You have [X] minutes. Prep for your next meeting with [Person]. Here's context..." |
| 15-30 min available | "You have [X] minutes. Quick calls only. Here are your top 3 contacts to call." → Routes to SALES MODE |
| 30+ min available | "You have [X] minutes. Let's get into sales mode." → Routes to SALES MODE |
| No meetings until afternoon | "Your morning is clear. This is call time. Let's go." → Routes to SALES MODE |

### Morning Protection Rule
- If Tony tries to schedule a meeting in the morning → system pushes back: "Mornings are for calls. Put this meeting in the afternoon."
- System will suggest afternoon time slots instead

### Meeting Warnings (Popups During Use)
- 30 min before: "You have a meeting with [Person] in 30 minutes. Make some quick calls or prep."
- 5 min before: "Meeting with [Person] starts in 5 minutes. Here's the context: [last meeting notes, what you promised]."

### Post-Meeting Processing
When a meeting ends (detected from calendar or Tony says "just finished"):
1. System pulls transcript from Plaud (see Section 9)
2. Extracts action items Tony committed to
3. Asks: "You just met with [Person]. You said you'd do these [X] things. Want me to create tasks?"
4. Asks: "Want me to schedule a follow-up?"
5. Tony can give additional instructions via voice
6. System cross-references against Tony's existing tasks and calendar: "You have more important things to do this week. You're behind on calls. Want to push this follow-up to two weeks from now?"
7. Updates contact record in master sales list with meeting notes and outcomes

# 17. INSTRUCTIONS SYSTEM (Every Element Has a Settings Icon)

### Purpose
Every feature, button, section, and element in the system has a small ⚙️ (gear/settings) icon. When Tony clicks it, a window opens where he can give custom instructions for that specific feature.

### How It Works
1. Tony clicks ⚙️ on any element (e.g., the Slack notification dot, the Ideas button, the Email popup)
2. A modal opens: "Instructions for [Feature Name]"
3. Tony speaks or types custom instructions: "I want this to also do X" or "Change this so it does Y instead of Z"
4. System evaluates the instruction and responds:
   - **YES:** "Got it. I updated [Feature Name]: [what changed]. This is now live."
   - **NO:** "I can't do that because [reason]. Here's what I CAN do instead: [alternative]."
5. If YES → the instruction is written to the relevant section of THIS MD file
6. The change log entry is added to the Training Log (Section 10) with type: INSTRUCTION_UPDATE

### What Tony Can Change via Instructions
- Email importance rules (add/remove senders, change keywords)
- Slack/Linear relevance rules (what counts as "relevant")
- Contact priority rules (who shows up first)
- Schedule rules (morning protection times, meeting duration defaults)
- Ideas categorization rules (add categories, change auto-prioritization logic)
- Accountability thresholds (how long before drift detection triggers)
- Tooltip content (update what a tooltip says)
- Any behavioral rule in this document

### What Tony CANNOT Change via Instructions
- The gate system (check-in is always required — non-negotiable)
- The accountability override → Ethan email rule (Tony can't disable being told on)
- The sales-first priority (sales calls always task #1)

### Logging
Every instruction change gets logged:
```
[DATE] INSTRUCTION_UPDATE: [Feature Name] — [What changed] — [Tony's exact words]
```

# 18. TOOLTIP SYSTEM (Every Element Has a Detailed Hover)

### Purpose
Every single element in the system has a hover-over tooltip. The tooltip explains EXACTLY what this element does, when it triggers, and the logic behind it. Detailed but as short as possible.

### Tooltip Structure
Each tooltip contains:
1. **What it is** — one sentence
2. **What it does** — what happens when you interact with it
3. **When it triggers** — under what conditions this becomes active/visible
4. **The logic** — why it works this way

### Tooltip Edit Button
Every tooltip has an **Edit ✏️** button at the bottom. When Tony clicks Edit:
1. The tooltip text becomes editable
2. Tony can modify the instructions/description
3. Changes save directly to this MD file under the relevant section
4. System acknowledges: "Updated tooltip for [element]. MD file updated."

This means Tony can modify system behavior directly from any tooltip — it's a shortcut to the Instructions System.

# 20. AGENT ARCHITECTURE

### Purpose
The system is designed as a set of specialized agents, each responsible for one domain. Each agent has its own instruction set (section in this MD file) and operates semi-independently while communicating through a shared context layer.

### Agent Roster

| Agent | Domain | MD Section | What It Owns |
|-------|--------|-----------|-------------|
| **Gate Agent** | Check-in + Journal | Sections 3.1, 3.2 | Morning gate, pattern alerts, journal formatting, Google Sheet/Doc updates |
| **Email Agent** | Email intelligence | Sections 3.3, 7 | Importance sorting, snooze management, reply drafting, thumbs up/down training, Gmail integration |
| **Calendar Agent** | Schedule + Meetings | Sections 3.4, 8 | Smart time routing, meeting warnings, contact pre-load, meeting history, post-meeting processing |
| **Sales Agent** | Sales Mode | Section 3.5 | Contact prioritization, call tracking, "Attempt Made" flow, script compliance, follow-up automation |
| **Task Agent** | Task Mode | Section 3.6 | OAP v4 priority enforcement, task tracking, "What Got Done" log |
| **Ideas Agent** | Ideas System | Section 4 | Idea capture, auto-prioritization against business plan, override detection, Ethan notification, Slack posting |
| **Slack Agent** | Slack Notifications | Section 5 (Slack) | Relevance filtering, smart escalation, reply from dashboard, dismiss/link |
| **Linear Agent** | Linear Notifications | Section 5 (Linear) | Due/overdue detection, blocker identification, Tony-mentioned issues |
| **Accountability Agent** | Accountability | Section 11 | Framework detection, drift detection, sales protection, EOD reports |
| **Transcript Agent** | Plaud / Recordings | Section 9 | Transcript processing, commitment extraction, task suggestion, meeting notes |

### Communication Between Agents
- All agents share access to this MD file as their instruction source
- All agents read/write to the Training Log (Section 10)
- All agents can trigger notifications to each other:
  - Calendar Agent → Sales Agent: "30 minutes free, route to sales mode"
  - Email Agent → Accountability Agent: "Tony replied to 0 important emails today"
  - Ideas Agent → Accountability Agent: "Tony overrode priority"
  - Sales Agent → Calendar Agent: "Schedule follow-up with [Person]"
  - Transcript Agent → Task Agent: "Tony committed to 3 action items in his last meeting"

### Implementation Note
Whether these run as separate Cowork scheduled tasks, separate Claude conversations, MCP servers, or modules within a single Replit app — Tony doesn't care. The MD file is the shared brain. Each agent reads its section, executes its domain, and writes results back. The architecture serves the function, not the other way around.
